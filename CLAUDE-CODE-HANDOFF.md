# WhatsApp ↔ Claude two-way assistant — build handoff

Two-way WhatsApp: I message my WhatsApp Business number and Claude replies; Claude
can also push messages to me. The reply is generated by a **Claude Code routine**
(trigger `trig_016Sm3srSGs8mk73hBVWWVJi`), not by the webhook.

## Architecture (and the one constraint that shapes it)
The routine `/fire` endpoint is **fire-and-forget** — it returns once the session is
created and never returns the assistant's output. So the reply can't ride back on the
HTTP response. Instead:

```
me → WhatsApp → Meta Cloud API ──webhook──► Cloud Function ──fire(text)──► routine
                                                                              │
me ◄── WhatsApp ◄── Meta Cloud API ◄── whatsapp-send skill ◄──── routine replies
```

The Cloud Function is dumb glue: verify → dedupe → fire. All intelligence + the reply
send live in the routine.

## Repo layout

```
functions/whatsapp-webhook/   # the GCP Cloud Function (gen2)
  index.js                    # single HTTP handler: GET verify + POST messages
  package.json
.claude/skills/whatsapp-send/ # skill the routine uses to send the reply
  SKILL.md
  whatsapp-send.sh            # curl POST to graph.facebook.com /messages
```

## Component 1 — the Cloud Function
Entry point `whatsapp`. Env vars:
- `VERIFY_TOKEN` — arbitrary string, also entered in Meta webhook config
- `APP_SECRET` — Meta App secret (verifies every POST's HMAC signature)
- `ROUTINE_FIRE_URL` — `https://api.anthropic.com/v1/claude_code/routines/trig_016Sm3srSGs8mk73hBVWWVJi/fire`
- `ROUTINE_TOKEN` — the routine's API-trigger bearer token (shown once at creation)
- `ALLOWED_WA_ID` — (optional) my own wa_id, so only I can trigger runs

Dedupe is Firestore-backed (collection `wa_dedupe`, doc per `message.id`): atomic
create-if-not-exists claim → fire → mark processed; release on failure so Meta retries.
A stale "processing" claim older than the lease is reclaimable (survives a crash mid-fire).

Deploy (pick a region near RO, e.g. europe-central2 = Warsaw). Keep secrets in
Secret Manager rather than plaintext `--set-env-vars`:

```bash
cd functions/whatsapp-webhook
gcloud functions deploy whatsapp-webhook \
  --gen2 --runtime=nodejs22 --region=europe-central2 \
  --source=. --entry-point=whatsapp \
  --trigger-http --allow-unauthenticated \
  --set-env-vars VERIFY_TOKEN=...,ALLOWED_WA_ID=40... \
  --set-secrets APP_SECRET=wa-app-secret:latest,ROUTINE_TOKEN=cc-routine-token:latest,ROUTINE_FIRE_URL=cc-routine-url:latest
```

`--allow-unauthenticated` is required (Meta calls it publicly); the HMAC signature
check is what actually secures it. The function's runtime service account needs the
`roles/datastore.user` role for Firestore. Then in Meta App → WhatsApp → Configuration,
set the callback URL to the function URL + verify token, and subscribe to `messages`.

## Component 2 — the whatsapp-send skill
The routine calls `./whatsapp-send.sh <reply_to_wa_id> "<text>"`. It reads from the
**routine's** environment (NOT this repo — don't commit these):
- `WHATSAPP_TOKEN` — permanent System User access token
- `WHATSAPP_PHONE_NUMBER_ID` — Cloud API phone number ID

## Routine configuration checklist (do this in claude.ai/code)
1. Point the routine at this repo (so it clones the `whatsapp-send` skill).
2. Environment variables: `WHATSAPP_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`.
3. **Allowed domains: add `graph.facebook.com`** — routines default to Trusted network
   access and it is NOT on the default allowlist; without it, sends fail with
   `403 host_not_allowed` silently.
4. Add the **Graphiti connector** to the routine — every fire is a fresh session with
   no history, so use Graphiti for cross-message conversation memory.
5. Routine prompt (starting point):
   > You handle incoming WhatsApp messages. The fired text gives `from_name`,
   > `reply_to_wa_id`, and `body`. Recall relevant context from Graphiti, do what the
   > body asks (you have your connectors and skills), then send exactly one reply by
   > running the `whatsapp-send` skill: `./whatsapp-send.sh <reply_to_wa_id> "<reply>"`.
   > Save anything worth remembering back to Graphiti.

## Gotchas
- **Network allowlist** for `graph.facebook.com` — routine setting, step 3 above.
- **Fresh-session amnesia** — solved via Graphiti, step 4.
- **24-hour window** — replies are fine (inside the window). Unprompted pushes outside
  24h of my last message get rejected (`131026`) and need an approved template.
